# @phloz/db

Drizzle schema, RLS policies, migrations, and seed for Phloz.

## Layout

```
src/
  client.ts              postgres.js + drizzle factories
  index.ts               public API
  schema/
    _helpers.ts          pkUuid, timestamps, userIdRef
    index.ts             barrel export (drizzle-kit reads this)
    <table>.ts           one file per table
  rls/
    _functions.sql       phloz_is_member_of, phloz_has_role_in, phloz_is_assigned_to
    <table>.sql          one policy file per tenant table
    index.ts             ordered list of SQL files + TENANT_TABLES registry
    apply.ts             CLI that runs every SQL file against DATABASE_URL
  seed/
    index.ts             minimal demo data
tests/
  rls/
    workspace-isolation.test.sql   pgTAP test covering the 3 required invariants
migrations/              generated by drizzle-kit (created by db:generate)
drizzle.config.ts
```

## Scripts

```bash
pnpm db:generate    # drizzle-kit generate — emits SQL migrations from src/schema
pnpm db:migrate     # drizzle-kit migrate — applies migrations to DATABASE_URL
pnpm db:push        # drizzle-kit push — dev-only, skips migration files
pnpm db:studio      # drizzle-kit studio — local GUI
pnpm db:seed        # runs src/seed/index.ts
```

RLS is not part of drizzle-kit's output. Run it separately:

```bash
tsx src/rls/apply.ts
```

Or roll it into `db:migrate` by wiring `apply.ts` into the post-migrate step.

## Adding a new tenant table

1. `src/schema/my-table.ts` — define the Drizzle table. Include `workspace_id
   uuid not null references workspaces(id) on delete cascade`.
2. Add the new file to `src/schema/index.ts` (alphabetical order).
3. `src/rls/my-table.sql` — enable RLS + add SELECT / INSERT / UPDATE / DELETE
   policies. Use the `phloz_is_member_of` / `phloz_has_role_in` helpers.
4. Add the SQL file to `RLS_FILES` in `src/rls/index.ts` and the table name
   to `TENANT_TABLES`.
5. `pnpm db:generate` + commit the generated migration.
6. Add a pgTAP test in `tests/rls/` if the policy is non-standard.

## RLS conventions

- Every tenant table enables RLS. The CI check reads `TENANT_TABLES` and
  queries `pg_tables.rowsecurity` to enforce this.
- Policies read `auth.uid()` (Supabase-provided). No hardcoded ids.
- Workspace-isolation is enforced on every SELECT through either
  `phloz_is_member_of(workspace_id)` or `phloz_is_assigned_to(client_id)`.
- Sensitive mutations (delete, role change) are pinned to
  `phloz_has_role_in(ws, ARRAY['owner', 'admin'])`.
- Service-role client bypasses RLS. Use it only in server code that has
  already performed its own auth check (webhooks, server actions, Inngest
  jobs).

See `skills/phloz-tenancy/SKILL.md` for the full playbook.

## Testing RLS locally

```bash
# in a Supabase local dev environment
supabase start
supabase db reset       # runs migrations + RLS
pg_prove -d postgres://postgres:postgres@localhost:54322/postgres tests/rls/*.sql
```
